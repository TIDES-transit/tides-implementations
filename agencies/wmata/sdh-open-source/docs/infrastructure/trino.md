# Trino

Trino is a distributed SQL query engine that provides users and tools with access to data stored in the Iceberg data lake. It connects to Lakekeeper as its Iceberg REST catalog and reads data files from ADLS Gen2.

## Architecture

The Trino module deploys two components:

- **Coordinator** (`[Project Name]-{env}-trino-ca`): Handles query planning, user authentication, and the Trino web UI
- **Workers** (`[Project Name]-{env}-trino-wrk-ca`): Execute query fragments and process data

Both communicate with Lakekeeper (for table metadata) and ADLS Gen2 (for data files). Workers discover the coordinator using its internal Container App hostname.

## Authentication Methods

Trino is configured to accept three types of authentication depending on the client:

| Method | Who uses it | How it works |
|--------|-------------|-------------|
| **OAuth2** | Human users via the Trino web UI | Azure AD login; users are redirected to sign in interactively |
| **JWT** | Machine-to-machine (e.g., Dagster pipelines) | Service principal access tokens passed as bearer tokens |
| **Password** | Legacy clients (e.g., Tableau, Metabase via JDBC) | Static username/password stored in Key Vault and hashed with bcrypt |

### Password Users

Password-based authentication is available for tools that cannot use OAuth2. Passwords are auto-generated by OpenTofu, stored in Key Vault as `trino-password-{username}`, and hashed with bcrypt for Trino's password file.

Current password users:

- `tableau` — for Tableau Server/Desktop JDBC connections

To add a new password user, add an entry to `local.trino_password_users` in `tf/modules/apps/trino/main.tf`.

## Access Control

Access control in Trino is enforced by Open Policy Agent (OPA). When a user runs a query, Trino asks OPA whether the action is permitted. OPA evaluates the request against `.rego` policy files that check the user's permissions in Lakekeeper.

This means Lakekeeper is the authoritative source for what data a user can access — the Lakekeeper role a user holds (via their Entra group membership) determines what they can query in Trino. See [Lakekeeper role management](./lakekeeper.md#role--permission-management) for how to manage user access.

## Entra Admin Consent (Required Before First Login)

The Trino app registration requests delegated permissions that require tenant-wide admin consent before any user can sign in. This is a one-time setup step.

**When `can_modify_entra = true`** (e.g., `consultant` environment): Admin consent is granted automatically by OpenTofu.

**When `can_modify_entra = false`** (e.g., `[env1]`, `[env2]`): A [AGENCY] Azure AD administrator must grant consent manually:

**Via the Azure Portal:**

1. Navigate to **Microsoft Entra ID** → **Enterprise applications**
2. Find **"Trino - [Project Name]"**
3. Go to **Permissions** → click **"Grant admin consent for [tenant]"**

**Via the admin consent URL:**

```text
https://login.microsoftonline.com/{tenant-id}/adminconsent?client_id={trino-client-id}
```

## Troubleshooting

### "Approval required" error on login

An administrator needs to grant tenant-wide admin consent for the Trino app registration. See [Entra Admin Consent](#entra-admin-consent-required-before-first-login) above.

### Workers not connecting to coordinator

Check that workers can resolve the coordinator's internal DNS name. Workers use `discovery.uri=http://{coordinator-app-name}` for service discovery within the Container App Environment.

### Lakekeeper catalog not accessible

Verify:

1. The `lakekeeper_catalog_url` variable is set to the Lakekeeper FQDN (not a hardcoded subdomain)
2. Trino's OAuth2 credentials have the Lakekeeper API scope assigned
3. The Lakekeeper Container App is running and healthy
